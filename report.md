1. 프로그램 주제와 목적

내 프로그램의 주제:
1인용 턴제 액션 RPG 게임

이 프로그램을 만든 이유:
파이썬 기초 문법(조건, 반복, 리스트, 함수)을 실제 동작하는 게임으로 구현하고, 평소에 게임을 좋아했기 때문에 간단하게 게임을 만들어보고싶었다

2. 문제 분해 및 모델링

입력

난이도: 쉬움('1') 또는 어려움('2') 선택

행운의 정수: 몬스터와 아이템의 무작위 배치를 연산할 임의의 수(시드값)

행동: 용사 조작을 위한 키 (W: 위, A: 왼쪽, S: 아래, D: 오른쪽, F: 공격, Q: 포기)

처리

수학적 난수 생성: 입력받은 정수를 수식에 적용해 가로(3~16), 세로(3~8) 구역 내부의 랜덤 좌표를 도출함

이동 및 벽 충돌 검사: 플레이어가 외각 벽 바깥(가로 1~18, 세로 1~10 구역 바깥)으로 움직이지 못하게 차단함

무필드 검 습득: 검 자리에 가면 턴 소모 없이 즉시 공격력을 올리고 맵에서 아이템을 제거함

사정거리 전투: 가로/세로 차이가 1칸 이내일 때만 공격(F)이 가능하게 하고 대미지를 정산함

3층 등반 시스템: 한 층의 몬스터 5마리를 모두 사냥하면 체력을 완치하고 다음 층 맵을 무작위로 구성함

출력

상태창: 현재 층수, 체력 게이지바([■■■■■□□□□□]), 현재 체력, 공격력, 사냥한 수 출력

격자 맵: 이중 반복문으로 12x20 크기의 격자에 플레이어(🧝), 몬스터(🟢, 🐷, 🍄, 🐗, 😈), 아이템(🗡️), 벽(#), 바닥(.)을 실시간 렌더링함

3. 사용한 2차원 리스트 설명

2차원 리스트 이름: monsters (몬스터 정보 데이터 리스트)

한 행의 의미: 사냥터에 등장하는 개별 몬스터들의 정보 (총 5행)

각 열의 의미:

열 번호

의미

실제 예시

0번 열

몬스터의 이름

"슬라임"

1번 열

몬스터의 체력 (HP)

2번 열

몬스터의 공격력 (ATK)

8

3번 열

몬스터의 현재 X좌표 (가로 위치)

3

(참고: 4번 열은 Y좌표, 5번 열은 생존 여부를 의미함)

이 2차원 리스트를 사용한 이유:
여러 마리의 몬스터들이 가진 다양한 종류의 데이터(이름, 체력, 공격력, 좌표)를 효율적인 하나의 묶음으로 구조화하여 일괄 관리하고 탐색하기 위해 사용했습니다.

5. 조건문과 반복문 활용

조건문 사용 부분

어떤 조건을 검사했는가?

if 1 <= next_x <= 18 and 1 <= next_y <= 10: 돌벽 외곽 테두리를 뚫고 나가지 못하게 이동 가드를 검사했습니다.

if abs(px - mx) <= 1 and abs(py - my) <= 1: 절댓값 연산식을 사용해 몬스터가 상하좌우 1칸 이내 사정거리에 있는지 확인했습니다.

if difficulty_choice == '1': 쉬움 모드일 때만 체력이 회복되도록 사용자가 선택한 난이도 상태를 비교 검사했습니다.

반복문 사용 부분

무엇을 반복 처리했는가?

while turn <= turns_limit and player['hp'] > 0: 생존 상태와 제한 턴 범위 내에서 주 게임 연산이 매끄럽게 반복 유지되도록 했습니다.

for row in map_template: 원본 맵 템플릿의 데이터 훼손을 방지하기 위해 한 줄씩 수동 깊은 복사를 수행하는 반복문을 가동했습니다.

for c in range(20) (중첩 for 루프): 12행 20열 격자판을 가로지르며 숫자를 해독해 예쁜 기호 및 이모지 이미지로 변환 인쇄했습니다.

조건문과 반복문이 함께 작동한 부분

중복을 막는 몬스터 좌표 배정:
while True 루프로 난수를 무한 생산하면서, 이미 좌표가 배정된 리스트를 for coord in occupied_coords문으로 순회하여 새로 추출한 좌표와 겹치는지 if문으로 정밀 대조 검사했습니다. 겹치지 않을 때만 안전하게 좌표를 확정하고 탈출하도록 제어 흐름을 복합 제어했습니다.

7. 테스트와 오류 수정

테스트 1: 정상 사례

입력: 플레이어가 W 키를 조작하여 검 아이템(🗡️) 칸을 밟음.

예상 결과: 공격력 변수가 오르고 맵 위에서 기호가 제거됨.

실제 결과: [아이템 획득] 기본칼을(를) 장착했습니다! 알림과 동시에 공격력이 15에서 20으로 가산되었으며 필드에서 안전 소멸했습니다.

테스트 2: 예외 또는 경계 사례

입력: 맵의 제일 서쪽 끝(x=1) 구역에 도달한 후 왼쪽으로 가기 위해 'A'를 억지로 타자 침.

예상 결과: 인덱스 경계 감도로 인해 이동이 막히고 경고문 표시.

실제 결과: 🛑 앗! 맵의 단단한 경계 벽에 막혀 움직일 수 없습니다!가 인쇄되며 제자리에 정차 고수되었습니다.

발견한 오류와 수정한 방법

오류: 층을 올라가면 플레이어가 장착했던 검 공격력 정보가 유실되어 다시 15 기본형 수치로 공중분해 되는 치명적인 초기화 버그가 있었습니다.

수정 방법: init_game() 함수에 existing_player=None 매개변수 구조를 수립하여, 승급 시 기존 플레이어 사전을 그대로 연계 인계하도록 보완해 체력만 만충하고 성장 ATK는 온전히 전이 보전시켰습니다.

8. Gemini 활용 내용 요약

Gemini에게 도움받은 부분:

random 모듈 배제 제약에 대응하는 수학적 난수 도식 생성(LCG 알고리즘) 아이디어 수립

2차원 리스트 복제 시 얕은 복사 훼손 결함을 막는 깊은 복사(list(row)) 수동 가이드 조언

Gemini가 제안했지만 내가 수정하거나 사용하지 않은 부분:

Gemini는 당초 몬스터를 맵 구석 돌벽 모서리 부근(가로 1~18, 세로 1~10)까지 빽빽하게 스폰시키는 넓은 연산을 추천했으나, 조작 시각 밸런스와 안전 공간 형성을 위해 안쪽 제한 영역(가로 3~16, 세로 3~8)으로 스폰 좌표 구역을 직접 타이트하게 수정하여 보강했습니다.

최종 코드에서 내가 직접 판단하거나 수정한 부분:

쉬움 모드('1')를 조작할 때 입문자들이 룰에 빨리 익숙해지도록, 몬스터의 기본 공격력을 절반(atk // 2)으로 영구 경감해 주는 밸런스 조정 판단 코드를 직접 기획해 덧붙였습니다.

9. 한계와 개선 방안

현재 프로그램의 한계:
텍스트 콘솔 기반 프로그램 조작 구조 때문에 방향 키를 한 번 입력할 때마다 Enter 키를 꾹 눌러야만 화면에 반영되어 다소 뻑뻑한 느낌이 듭니다.

다시 개선한다면 추가하고 싶은 기능:
실시간 입력 추적 라이브러리 모듈을 활용하여 엔터 조작을 영구 배제하고, 화살표 방향키 터치만으로 2차원 공간을 매끄럽게 돌아다니는 조작감 보완을 이루고 싶습니다.

10. 코드인터뷰 준비

1. 내 프로그램의 입력-처리-출력은 무엇인가?

입력: 난이도('1' 또는 '2'), 시드값(임의의 정수), 용사 조작 키 지시어('WASDFQ')

처리: 수학식을 활용한 중복 없는 무작위 생성, 통행 범위 제어, 검 실시간 무필드 줍기, 절댓값 사정거리 마찰, 층 승급 연계

출력: 현재 층수 및 게이지바가 표기된 상단 정보바와 12x20 이모지 실시간 경기장

2. 2차원 리스트에서 한 행과 각 열은 무엇을 의미하는가?

한 행(Row)은 맵의 가로 Y축 라인을 뜻하며 0번부터 11번까지 12줄이 존재합니다.

각 열(Column)은 맵의 세로 X축 칸을 뜻하며 0번부터 19번까지 20칸이 존재합니다.

(몬스터 데이터 사전 monsters에서는 행이 각각의 마수, 열은 0:이름, 1:HP, 2:ATK, 3:X좌표를 표상합니다.)

3. 가장 중요한 함수는 무엇이고 어떤 역할을 하는가?

init_game(difficulty_choice, lucky_seed, existing_player)입니다.

새로운 던전 층의 로딩과 리셋 데이터를 총합 정리하는 핵심 허브이며, 특히 existing_player 인계 연동을 통해 플레이어가 앞서 수확해 둔 검 공격력 정보를 버리지 않고 온전히 다음 층 사냥터로 승계하게 돕는 징검다리 역할을 수행합니다.

4. 조건문과 반복문은 어디에서 사용되었는가?

조건문: 가로 1~18, 세로 1~10 충돌 제한 가드, 절댓값 사정거리 타격 검출(abs), 쉬움 모드 처치 버프에 사용되었습니다.

반복문: 생존 한계까지 주 모험 루틴을 유지하는 while 주 루프, 수동으로 1차원 줄을 쪼개 복사하는 깊은 복사 for 루프, 12x20 전체 공간의 기호를 번역해 출력하는 중첩 for 루프에 사용되었습니다.

5. Gemini가 도와준 부분과 내가 직접 수정한 부분은 무엇인가?

도움받은 부분: 라이브러리 우회 수학적 난수 연산 설계(LCG) 협업 및 메모리 주소 공유에 기인한 원본 지도 파괴 예외 자문

직접 수정한 부분: 스폰 범위 안쪽(3~16, 3~8)으로 보정 축소, 쉬움 난이도 선택 시 몬스터 위협 타격 수치를 강제 절반 하향(atk // 2)하여 밸런싱을 조정한 부분입니다.
