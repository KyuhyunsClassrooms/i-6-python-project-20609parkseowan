## 3차 Gemini 활용

사용 날짜: 6/15

사용 목적: 프로그래밍 오류 확인 및 수정사항 (층 이동 시 캐릭터 성장 공격력 초기화 문제 디버깅)

내가 입력한 프롬프트:
"층이 이동할 때 캐릭터가 먹었던 검 아이템이나 공격력 수치가 리셋되어 1층 초기 상태로 돌아가 버려. 1층에서 획득한 스탯과 검 장착 데이터를 유지하면서 다음 층으로 넘어가게 하는 방법을 알려줘"
Gemini가 제안한 내용 요약: init_game() 함수 인자에 기본값을 설정하는 existing_player=None 매개변수 구조를 전수하고, 층 승급 호출부에서 이전 층의 캐릭터 딕셔너리(player)를 직접 인수값으로 전달하여 동일한 메모리를 전이 공유하게 하라고 제안함.

내가 반영한 부분: init_game 함수의 인자 리스트 및 분기 구조 보완 완료
내가 수정하거나 사용하지 않은 부분: Gemini는 체력과 처치 수까지 원본 데이터를 고스란히 살려 가라고 했으나, 다음 층의 밸런스 난이도 억제 및 등반의 가치를 살리기 위해 체력을 가득 채우고(hp = max_hp) 현재 층 처치 수(kills)만은 0으로 온전히 초기화하는 분기 조건을 스스로 보강하여 코딩을 작성함.

반영 위치: init_game() 함수의 내부 구현 파트 및 main() 주 루프의 층 등반 조건문(if player['kills'] == 5) 블록 구역

## AI 활용 성찰

Gemini가 도움이 된 점:
단순 코드 구현을 넘어 원리 파악을 위해 지속적으로 피드백을 주고받는 등 AI를 주도적이고 능동적인 협력 학습 도구로 활용하는 방법과 비슷한 바이브코딩에 대해 조금 더 알아가게됨

Gemini의 제안을 그대로 쓰지 않고 내가 판단하거나 수정한 점:
외부 라이브러리(random)을 사용하려했기에 나는 if로 바꿔서 다른 경우들을 배치함
몬스터들의 시작 스폰 영역이 맵 벽면 가장자리에 타이트하게 붙는 것이 플레이 환경 상 조잡해 보여, 스폰 경계 범위를 축소한 내부 범위(가로 3~16, 세로 3~8)로 직접 제한 수치를 기획하여 수정했다 또한, 쉬움 모드의 밸런스를 잡기 위해 몬스터의 공격력을 절반(atk // 2)으로 약하게 조정해 주는 판단 코드를 스스로 설계해 삽입하였습니다.

다음에 AI를 사용할 때 주의해야 할 점:
AI가 제안하는 코드는 매우 정교하지만 가끔 복잡하고 비효율적인 고급 클래스 객체나 우리가 정식으로 학습하지 않은 외부 라이브러리 연산을 무분별하게 유도할 수 있었다. 따라서 AI의 조언을 비판적으로 검토하지 않고 맹목적으로 복사해 붙여넣으면 오류가 나거나 학교 제출 시 설명 능력을 증명할 수 없게 된다. 앞으로는 배운 기본 문법의 테두리를 항상 의식하고, "내가 스스로 완벽하게 줄단위 주석으로 해설하고 면접에 응할 수 있는 코드인가"를 스스로 자가 점검하는 습관을 키워야겠다고 생각했다"
